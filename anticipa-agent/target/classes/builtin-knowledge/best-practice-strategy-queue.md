# 队列满时的处理策略

## 队列满的原因
- 瞬时流量激增超过系统容量
- 消费者（线程池）处理能力不足
- 队列容量设置过小
- 生产者的生产速度持续 > 消费者的消费速度

## 应对策略

### 方案 A：扩容
增加 maximumPoolSize 或队列容量，提高系统吞吐能力。

**优点**：从根本上解决问题
**缺点**：不能无限扩容，受限于硬件资源

### 方案 B：降级
拒绝非核心任务，优先保证核心业务。使用 `CallerRunsPolicy` 让调用者自己执行。

**优点**：实现简单，天然背压
**缺点**：调用者线程被阻塞，可能连锁反应

### 方案 C：限流
在入口处限流，避免冲击线程池。

```java
// 结合 Sentinel 或 RateLimiter
if (!rateLimiter.tryAcquire()) {
    // 返回降级结果
    return fallback();
}
```

**优点**：保护系统不被冲垮
**缺点**：需要额外组件

### 方案 D：延迟处理
将无法处理的任务写入 MQ 或 DB，后续异步补偿。

**优点**：不丢失任务
**缺点**：引入额外依赖，增加复杂度

## 决策流程
```
队列是否持续满？
  ├── 是瞬时峰值 → 增加队列容量（方案A）
  ├── 持续满，但总体负载可接受 → 增加线程数（方案A）
  ├── 持续满且负载已高 → 限流 + 降级（方案B+C）
  └── 不能丢任务 → 写延迟队列（方案D）
```

## 注意事项
- 不要单纯依赖增大队列容量，这只会推迟问题暴露的时间
- 队列满通常是系统容量不足的信号，需要从根本扩容
- 结合 Anticipa 的监控告警，队列使用率 > 80% 时应自动告警
