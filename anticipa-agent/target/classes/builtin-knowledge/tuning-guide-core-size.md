# 核心线程数调参指南

## 确定基准值的方法

### 方法 1：经验公式（起点）
- **CPU 密集型**：`corePoolSize = CPU核数 + 1`
- **IO 密集型**：`corePoolSize = CPU核数 × 2`
- **混合型**：`corePoolSize = CPU核数 × 1.5 ~ 2`

### 方法 2：压测确定（推荐）
1. 从经验公式的基准值开始
2. 以固定 QPS 压测，逐步增加线程数
3. 观察吞吐量和响应时间的变化
4. 找到**吞吐量拐点**对应的线程数

```
吞吐量
  ↑
  |     ┌──── 最佳值
  |     │
  |    ╱╲     ← 超过此点，线程过多导致上下文切换
  |   ╱  ╲     成本 > 新增线程带来的收益
  |  ╱    ╲
  | ╱      ╲
  └──────────────────→ 线程数
```

### 方法 3：自适应
利用 IO 时间与 CPU 时间的比例计算：
```
线程数 = CPU核数 × (1 + 等待时间 / 计算时间)
```
例如：CPU 计算 100ms，IO 等待 300ms
→ 线程数 = 16 × (1 + 300/100) = 64

## 调整步骤
1. **初始设置**：按经验公式配置
2. **运行观察**（1~3 天）：监控活跃线程数 + 队列使用率
3. **微调**：基于实际数据调整 corePoolSize
4. **验证**：调整后继续观察 1~3 天
5. **固化**：找到稳定值后固定配置，通过 Anticipa 调整

## 注意事项
- 每次调整幅度建议 20%~30%，不宜一次调整过大
- 调整后至少观察一个业务周期（如一天的高峰期）
- 同时监控上下游（数据库连接池、下游服务）是否被打满
